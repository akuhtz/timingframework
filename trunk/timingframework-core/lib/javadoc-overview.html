The Timing Framework is a set of utility classes that enables you to create powerful animations in your Java applications.

The Timing Framework was implemented by Chet Haase and introduced in the book Filthy Rich Clients (Haase and Guy, Addison-Wesley, 2008).

Improvements to Haase's release were implemented by Tim Halloran beginning in 2011. These improvements include

  • A separation of generic Java code from Swing-specific code. This allows the Timing Framework to be useful for a wide-variety of graphical interface libraries. Two are currently supported: Swing and SWT. The original implementation was, in places, quite Swing-specific. Generic Java code is placed in packages under org.jdesktop.core and released (stand-alone) as the timingframework-core distribution at java.net.
  • Improved support for the AWT/Swing graphical interface library. Swing-specific code is placed in packages under org.jdesktop.swing and released (combined with the core code) as the timingframework-swing distribution at java.net.
  • Support for the SWT graphical interface library. A competitor to Swing that is used by the popular Eclipse platform. SWT-specific code is placed in packages under org.jdesktop.swt and released (combined with the core code) as the timingframework-swt distribution at java.net.
  • Use of the builder pattern to construct animations. This avoids the mix of passing some parameters to an Animator constructor and some to "set" methods. For example, The following animation will run a five second animation four times on myTarget: 
    Animator animator = new AnimatorBuilder().setDuration(5, TimeUnit.SECONDS)
                                         .setRepeatCount(4)
                                         .addTarget(myTarget)
                                         .build();
animator.start();
  • The ability to support a single timing source as advocated by Chet Haase in the JavaOne 2008 talk Filthy Rich Clients: Filthier Richer Clientier. This is done by setting a default timing source on the AnimatorBuilder class. For example, the following snippet of code creates a Swing timer and sets it as the default for all animations. 
    TimingSource source = new SwingTimerTimingSource(15, TimeUnit.MILLISECONDS);
AnimatorBuilder.setDefaultTimingSource(source); // shared timing source
source.init(); // starts the timer
...
// create, use, and dispose lots of Animator instances
...
source.dispose(); // stops the timer
  • A multi-thread safe implementation that takes advantage of the util.concurrent library.

The Swing and SWT releases contain several demonstration programs that can be run from the release Jar file or built from source code. Many of these demonstration programs are updated versions of the programs presented in Filthy Rich Clients and they are organized by chapter, e.g., ch14 for Chapter 14. Each demonstration is described in the book and within its source code.

The Swing demonstrations are:

  • org.jdesktop.swing.animation.timing.demos.ch12.TimingSourceResolution
  • org.jdesktop.swing.animation.timing.demos.ch14.FadingButtonTF
  • org.jdesktop.swing.animation.timing.demos.ch14.BasicRace
  • org.jdesktop.swing.animation.timing.demos.ch14.NonLinearRace
  • org.jdesktop.swing.animation.timing.demos.ch14.SplineInterpolatorTest
  • org.jdesktop.swing.animation.timing.demos.ch15.Triggers
  • org.jdesktop.swing.animation.timing.demos.ch15.ClickAndGo (not in the book—demonstrates non-trivial property setters)
  • org.jdesktop.swing.animation.timing.demos.ch15.DiscreteInterpolation (console)
  • org.jdesktop.swing.animation.timing.demos.ch15.MultiStepRace
  • org.jdesktop.swing.animation.timing.demos.javaone2008.TooManyBallsBroken (illustrates the Swing timer issues Chet Haase discussed at JavaOne 2008)
  • org.jdesktop.swing.animation.timing.demos.javaone2008.TooManyBalls (fixed version—this demonstration is the Jar's Main-Class)

To run one of the demonstrations from the release Jar file use:

java -cp timingframework-swing-2.0.jar classname

The Swing TooManyBalls demonstration can be changed from passive rendering to game-like active rendering (see the JActiveRenderer class) by defining the org.jdesktop.renderer.active property. For example, use:

java -cp timingframework-swing-2.0.jar -Dorg.jdesktop.renderer.active=true org.jdesktop.swing.animation.timing.demos.javaone2008.TooManyBalls

On some computers active rendering looks smoother, but has the cost of using two processors and lots of memory. It is perhaps more appropriate for a game than a desktop application. It is, however, straightforward to write programs that can change from passive to active rendering, like TooManyBalls does, by implementing JRendererTarget.

The Swing distribution also includes org.jdesktop.swing.animation.timing.splineeditor.SplineEditor (from Chapter 14) which can be used to design and preview a particular spline interpolation.

The SWT demonstrations are:

  • org.jdesktop.swt.animation.timing.demos.ch12.TimingSourceResolution
  • org.jdesktop.swt.animation.timing.demos.ch14.FadingButtonTF (fades the background not the button because SWT doesn't support transparent components)
  • org.jdesktop.swt.animation.timing.demos.ch14.BasicRace
  • org.jdesktop.swt.animation.timing.demos.ch14.NonLinearRace
  • org.jdesktop.swt.animation.timing.demos.ch14.SplineInterpolatorTest
  • org.jdesktop.swt.animation.timing.demos.ch15.Triggers
  • org.jdesktop.swt.animation.timing.demos.ch15.ClickAndGo (not in the book—demonstrates non-trivial property setters)
  • org.jdesktop.swt.animation.timing.demos.ch15.DiscreteInterpolation (console)
  • org.jdesktop.swt.animation.timing.demos.ch15.MultiStepRace
  • org.jdesktop.swt.animation.timing.demos.javaone2008.TooManyBalls (SWT version—this demonstration is the Jar's Main-Class)

To run one of the SWT demonstrations from the release Jar file use:

java -cp "timingframework-swt-2.0.jar;swt.jar" classname

In the example above I copied the swt.jar for my platform into the same directory, but you can reference it within another directory (the command above worked on Windows under Cygwin).

Neither the spline editor nor active rendering are currently implemented under SWT.
